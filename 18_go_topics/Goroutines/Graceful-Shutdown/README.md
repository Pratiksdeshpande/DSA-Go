# 1. Introduction

Graceful Shutdown is the process of safely terminating an application while ensuring:

* No requests are lost
* No messages are abandoned
* No partial database writes occur
* No goroutines are leaked
* No resources remain open

A graceful shutdown allows the system to transition from:

```text
RUNNING
   ↓
STOPPING
   ↓
CLEANUP
   ↓
TERMINATED
```

instead of immediately killing the process.

---

# 2. What is Graceful Shutdown?

Graceful Shutdown means:

> Stop accepting new work, finish ongoing work, release resources, and then exit.

---

## Bad Shutdown

```text
SIGTERM
   ↓
Process exits immediately
```

Problems:

* Active requests dropped
* DB transactions interrupted
* Messages lost
* Goroutines killed

---

## Graceful Shutdown

```text
SIGTERM
   ↓
Stop accepting new work
   ↓
Cancel background tasks
   ↓
Finish active work
   ↓
Close resources
   ↓
Exit
```

---

# 3. Why Graceful Shutdown Matters

Consider:

```text
HTTP Request
    ↓
Write Database
    ↓
Publish Event
```

Now pod receives SIGTERM.

```text
Database Updated
Event Not Published
```

System becomes inconsistent.

Graceful shutdown exists to avoid these situations.

# 5. OS Signals

---

## SIGINT

Generated by:

```bash
Ctrl+C
```

Purpose:

```text
User requests interruption
```

---

## SIGTERM

Purpose:

```text
Request application to terminate gracefully
```

Commonly used by:

* Kubernetes
* Docker
* systemd

---

## SIGKILL

Purpose:

```text
Force immediate termination
```

Cannot be intercepted.

Cannot be handled.

Cannot be ignored.

```text
SIGKILL
    ↓
Immediate Death
```

# 6. How Go Handles Signals

Go provides:

```go
signal.Notify(...)
```

Example:

```go
sigCh := make(chan os.Signal, 1)

signal.Notify(
    sigCh,
    syscall.SIGINT,
    syscall.SIGTERM,
)
```

Flow:

```text
OS Signal
    ↓
Go Runtime
    ↓
Signal Channel
    ↓
Application Logic
```

# 7. Context Cancellation

Context is the standard mechanism for coordinating shutdown.

Example:

```go
ctx, cancel := context.WithCancel(
    context.Background(),
)
```

Shutdown:

```go
cancel()
```

Internally:

```text
cancel()
    ↓
ctx.Done() closes
    ↓
all listeners notified
```

Workers:

```go
select {
case <-ctx.Done():
    return
}
```

This creates a system-wide shutdown signal.

---

# 8. WaitGroups During Shutdown

Problem:

```go
go worker()

return
```

Main exits.

Worker dies abruptly.

---

Solution:

```go
var wg sync.WaitGroup

wg.Add(1)

go worker()

wg.Wait()
```

Flow:

```text
Shutdown Signal
      ↓
Cancel Context
      ↓
Workers Exit
      ↓
wg.Done()
      ↓
wg.Wait() Unblocks
      ↓
Process Exits
```

# 10. HTTP Server Graceful Shutdown

Bad:

```go
http.ListenAndServe(":8080", nil)
```

Process termination drops active requests.

---

Good:

```go
server := &http.Server{
    Addr: ":8080",
}

go server.ListenAndServe()

ctx, cancel := context.WithTimeout(
    context.Background(),
    30*time.Second,
)

defer cancel()

server.Shutdown(ctx)
```

Internally:

```text
Stop New Connections
        ↓
Allow Existing Requests
        ↓
Wait For Completion
        ↓
Close Listener
        ↓
Exit
```

# 13. Kubernetes Graceful Shutdown

Kubernetes termination sequence:

```text
Pod Running
      ↓
SIGTERM
      ↓
terminationGracePeriodSeconds
      ↓
SIGKILL
```

Example:

```yaml
terminationGracePeriodSeconds: 30
```

Application must finish shutdown before:

```text
30 Seconds
```

otherwise:

```text
SIGKILL
```

is issued.

# 14. Common Production Mistakes

---

## Mistake 1

Ignoring Context

```go
for {
    process()
}
```

Worker never exits.

---

## Mistake 2

Not Waiting For Goroutines

```go
go worker()

return
```

Worker abruptly dies.

---

## Mistake 3

Closing Channel Multiple Times

```go
close(ch)
close(ch)
```

Causes panic.

---

## Mistake 4

Waiting Forever

```go
wg.Add(1)

// forgot Done()
```

Shutdown hangs forever.

---

## Mistake 5

Long Running Requests

A request may take:

```text
10 minutes
```

Shutdown never finishes.

Need:

```go
context.WithTimeout(...)
```

# 17. L5 Interview Questions and Answers

---

## Q1. What is Graceful Shutdown?

### Answer

Graceful shutdown is the controlled termination of an application where the system stops accepting new work, allows in-flight operations to complete, releases resources, and exits safely without losing data or corrupting state.

---

## Q2. Why is graceful shutdown critical in distributed systems?

### Answer

Distributed systems often execute multi-step workflows.

Example:

```text
Write Database
Publish Event
Update Cache
```

If termination occurs in the middle, system consistency may be violated.

Graceful shutdown helps complete these operations before exiting.

---

## Q3. Why is context preferred over boolean flags?

### Answer

Boolean flags do not support:

* propagation
* deadlines
* cancellation trees
* integration with standard libraries

Context provides:

```text
Cancellation
Deadlines
Timeouts
Propagation
```

in a standardized way.

---

## Q4. What is the role of WaitGroup during shutdown?

### Answer

WaitGroup guarantees that all goroutines finish execution before the main goroutine exits.

Without WaitGroup:

```text
Main exits
    ↓
All goroutines die
```

potentially losing work.

---

## Q5. What happens internally when cancel() is called?

### Answer

Internally:

```text
cancel()
     ↓
Done channel closes
     ↓
All waiters awakened
     ↓
Goroutines observe cancellation
     ↓
Shutdown begins
```

The Done channel acts as a broadcast signal.

---

## Q6. Why can channel closing be used for shutdown?

### Answer

When a channel is closed:

```go
<-ch
```

immediately unblocks all receivers.

This creates a natural broadcast mechanism.

Example:

```go
close(stopCh)
```

All workers receive shutdown notification.

---

## Q7. Explain Kubernetes graceful shutdown.

### Answer

Kubernetes sends:

```text
SIGTERM
```

Application gets a grace period.

During that period:

* stop accepting traffic
* finish active work
* release resources

If not completed:

```text
SIGKILL
```

is issued.

---

## Q8. How would you gracefully shutdown a worker pool?

### Answer

```text
Receive SIGTERM
      ↓
Stop producers
      ↓
Close intake
      ↓
Drain queue
      ↓
Workers finish
      ↓
WaitGroup wait
      ↓
Exit
```

---

## Q9. How would you gracefully shutdown an HTTP server?

### Answer

Use:

```go
server.Shutdown(ctx)
```

which:

```text
Stops new connections
Allows existing requests
Waits for completion
Closes listener
Returns
```

---

## Q10. What are common shutdown bugs in production?

### Answer

* Ignoring context cancellation
* Forgetting WaitGroup Done
* Leaking goroutines
* Not closing resources
* Infinite retries during shutdown
* Blocking cleanup logic
* Using SIGKILL instead of SIGTERM
* Unbounded shutdown duration

---

## Q11. How do you debug a service that hangs during shutdown?

### Answer

Investigate:

```text
Blocked Goroutines
WaitGroup Deadlocks
Context Ignored
Channel Waits
Mutex Contention
Long-running Requests
```

Common tools:

```bash
kill -QUIT <pid>
```

```bash
go tool pprof
```

```bash
go tool trace
```

Goroutine dumps often reveal the blocking operation.

---

## Q12. What is the most important principle of graceful shutdown?

### Answer

The most important principle is:

> Stop accepting new work before waiting for existing work to complete.

If new work continues arriving, shutdown may never finish.

---

# 18. Key Takeaways

* Graceful shutdown prevents data loss and state corruption.
* SIGTERM is the standard termination signal.
* SIGKILL cannot be intercepted.
* Context is the preferred cancellation mechanism.
* WaitGroups ensure goroutines finish before exit.
* HTTP servers should use `server.Shutdown()`.
* Worker pools should drain existing work.
* Kubernetes sends SIGTERM before SIGKILL.
* Shutdown paths must be tested like normal business logic.
* Graceful shutdown is a correctness problem, not merely an operational concern.
